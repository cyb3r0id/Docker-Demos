🏠 Docker Host, Daemon & Client ▼
Docker Host is the machine where the Docker Engine is installed. It can be a physical machine, a virtual machine, or even your local desktop.
1 Docker Daemon (dockerd)
Runs on the Docker host and listens for Docker API requests. It manages Docker objects — images, containers, networks, and volumes.
2 Docker Client (docker CLI)
The primary way users interact with Docker. It can live on the Docker host itself, or on a completely separate machine (like your laptop) and connect remotely to the host. When you run docker images or docker run, the client sends the command to the daemon, which carries it out via the Docker API.
3 One Client, Many Daemons
A single Docker client can communicate with more than one daemon — if Docker is installed on multiple hosts, you can point the client at a specific host and run commands against it.
Diagram 1 — The client sends commands over the Docker API; the daemon executes them and manages images, containers, networks, and volumes on the host.
ℹ️
The CLI Name Is "docker"
Type docker images in a terminal and it lists your images — that command is the Docker client talking to the daemon.
📦 Docker Images ▼
Definition: A Docker image is a read-only template with instructions for creating a Docker container — similar to how a virtual machine image is used to create a VM.
ℹ️
Images Are Often Based on Other Images
An image is frequently built on top of another image, with extra customization. Example: to add custom configuration to an nginx web server, take the base nginx image from Docker Hub and modify it — typically with a FROM nginx line in a Dockerfile, then layer your own customizations on top.

🧊 Docker Containers ▼
Definition: A Docker container is a runnable instance of an image. Once you have an image, you run it as a container — the live process or application runs inside, and you access your app through the container's endpoint.
Diagram 2 — Each container is a separate runnable instance of the same image, managed via the Docker API or CLI.
ℹ️
What You Can Do From a Container
Create, start, stop, move, or delete a container using the Docker API or CLI. You can also connect a container to one or more networks, attach storage, and even create a new image based on its current state.
⚠️
Containers Are Ephemeral by Default
When a container is removed, any changes to its state that aren't stored in persistent storage disappear. By default a container only has ephemeral (temporary) storage — stop the container, and the data is gone unless persistent storage is attached. Persistent storage is covered practically through volumes, bind mounts, and tmpfs mounts in later demos.
☁️ Docker Registry & Docker Hub ▼
Definition: A Docker registry stores Docker images. Docker Hub is a public registry that anyone can use, and Docker is configured to look there by default. You can also run your own private registry.
Diagram 3 — Build a Docker image locally, push it to Docker Hub (or a private registry), then pull and run it from anywhere.
ℹ️
Push vs Pull

docker push — uploads an image you built locally to the configured registry (e.g. Docker Hub).

docker pull / docker run — downloads the required image from the configured registry, ready to run as a container.

📋 Terminology Summary ▼
TermWhat It Is
Docker HostThe machine (physical, VM, or local desktop) where the Docker Engine is installed.
Docker Daemon (dockerd)Runs on the host, listens for API requests, and manages images, containers, networks, and volumes.
Docker Client (docker)The CLI you use to send commands to the daemon — can run on the host or connect remotely.
Docker ImageA read-only template with instructions for creating a container; often based on another image.
Docker ContainerA runnable instance of an image; ephemeral storage by default unless persistent storage is attached.
Docker Registry / Docker HubA central repository that stores Docker images; Docker Hub is the public default, private registries are also possible.
✅
Summary — The Core Mental Model

Host = where Docker runs. Daemon = the engine that does the work. Client = how you talk to it.

Image = the read-only blueprint. Container = a live, running instance of that blueprint.

Registry / Docker Hub = where images are stored centrally, so they can be pushed once and pulled anywhere.
