🏷️ What Is the LABEL Instruction? ▼
Definition: LABEL adds metadata to a Docker image. An image can carry more than one label.
1 Labels are inherited from the base image
Whatever labels exist in your FROM base image are carried forward into your custom image automatically — just like the earlier demo where FROM nginx brought nginx's own image contents along with it.
2 Labels can be overridden
If a label already exists (inherited from the base image) but you set it again with a different value, the most recently applied value wins at build time.
📐 Custom Labels vs OCI Labels ▼
Custom Labels
🏷️ Simple key=value pairs you make up yourself
⚠️ maintainer is a common one, but is deprecated
📄 e.g. version, description
OCI Labels (recommended)
✅ Standardized under org.opencontainers.image.*
📋 author, title, description, version, revision, created, url, source, documentation, vendor, licenses
🌍 Understood consistently across tools & registries
ℹ️
Why OCI Labels Are Preferred
maintainer is still widely used out of habit, but it's deprecated. The recommended approach is the Open Container Initiative (OCI) label set — a standardized naming convention every OCI-compliant tool understands the same way.
🧱 The Dockerfile — Labels Demo ▼
The base image for this demo is nginx:alpine-slim — a much smaller variant of nginx than the plain nginx image used in the previous demo.
Dockerfile
FROM nginx:alpine-slim

# Custom labels
LABEL maintainer="yourname@example.com"
LABEL version="1.0"
LABEL description="Demo image for Dockerfile LABEL instruction"

# Standard OCI (Open Container Initiative) labels
LABEL org.opencontainers.image.authors="Your Name"
LABEL org.opencontainers.image.title="Demo4 Dockerfile Labels"
LABEL org.opencontainers.image.description="nginx alpine-slim image with custom and OCI labels"
LABEL org.opencontainers.image.version="1.0"
LABEL org.opencontainers.image.revision="1"
LABEL org.opencontainers.image.created="2026-07-13"
LABEL org.opencontainers.image.url="https://github.com/stacksimplify/my-nginx"
LABEL org.opencontainers.image.source="https://github.com/stacksimplify/my-nginx/Dockerfile"
LABEL org.opencontainers.image.documentation="https://github.com/stacksimplify/my-nginx/README.md"
LABEL org.opencontainers.image.vendor="StackSimplify"
LABEL org.opencontainers.image.licenses="Apache-2.0"

COPY index.html /usr/share/nginx/html
ℹ️
What Each OCI Field Means

authors — who wrote it. title / description — what it is. version / revision — which build. created — build date.

url / source — where to find the project / Dockerfile. documentation — where to read more. vendor — who publishes it. licenses — under what license.

🔨 Build & Run the Image ▼
1 Clean up first (good habit between demos)
bash
docker rm -f myapp1
docker rmi $(docker images -q)
This removes any leftover container and every local image, giving you a clean slate for this demo.
2 Build the image
bash
docker build -t demo4-dockerfile-labels:v1 .
3 Compare the image size
bash
docker images
ImageSize
nginx (full)~279 MB
demo4-dockerfile-labels:v1 (nginx:alpine-slim)~19.2 MB
💡
Image Size Matters
Choosing an alpine-slim base instead of the full nginx image cuts the image size dramatically — from ~279 MB down to ~19.2 MB — because it strips out tooling the app doesn't actually need. Choose your base image based on what your application really requires.
4 Run a container from it
bash
docker run --name my-labels-demo -p 8080:80 -d demo4-dockerfile-labels:v1
Visit http://localhost:8080 to confirm the custom index.html loads.
🔍 Verifying Labels — docker image inspect ▼
1 Full image inspect
bash
docker image inspect demo4-dockerfile-labels:v1
Returns a large JSON document: architecture (amd64), OS (linux), and — scrolling up — a Labels section listing every label you set: description, maintainer, image authors, created date, documentation, and more.
2 Print only the labels
Use --format to filter down to just the labels section:
bash
docker image inspect --format '{{json .Config.Labels}}' demo4-dockerfile-labels:v1
3 Pretty-print with jq (optional)
The raw JSON from the previous command isn't pretty-printed. Pipe it through jq for readable, indented output:
bash
docker image inspect --format '{{json .Config.Labels}}' demo4-dockerfile-labels:v1 | jq
jq is entirely optional — it just formats the JSON nicely. Installation steps differ by OS (macOS, Debian/Linux, Windows) and are provided separately in the course reference material; it isn't required to complete this demo.
4 Get just the creation date
bash
docker image inspect --format '{{.Created}}' demo4-dockerfile-labels:v1
📦 Verifying the Container — docker inspect ▼
Labels aren't only on the image — since the container was created from that image, the same labels show up when you inspect the container too.
1 Full container inspect
bash
docker inspect my-labels-demo
Returns detailed JSON: DNS names, IP address, gateway, MAC address, mounted volumes, labels, and much more.
2 Get just the container's IP address
bash
docker inspect --format '{{.NetworkSettings.IPAddress}}' my-labels-demo
3 Get just the container's status
bash
docker inspect --format '{{.State.Status}}' my-labels-demo
4 Get the exposed ports
bash
docker inspect --format '{{json .Config.ExposedPorts}}' my-labels-demo
Since the container port used was 80, that's the value you'll see here.
5 Get full network settings
bash
docker inspect --format '{{json .NetworkSettings}}' my-labels-demo
CommandInspects
docker image inspect <image>An image — its labels, layers, architecture, and config.
docker inspect <container>A container — its network settings, state, mounts, and inherited labels.
🧹 Cleanup & Summary ▼
1 Remove the container
bash
docker rm -f my-labels-demo
2 Remove the image
bash
docker rmi demo4-dockerfile-labels:v1
Leaving the local desktop clean and ready for the next demo.

Summary

LABEL attaches metadata to an image; labels inherit from the base image and later values override earlier ones.

Prefer standardized OCI labels (org.opencontainers.image.*) over ad-hoc custom labels like the deprecated maintainer.

docker image inspect verifies labels on the image; docker inspect verifies them (and much more) on the running container.
