🎯 Learning Objectives ▼
💡
By the end of this demo, you should be able to:

1. Explain the "fetch from URL" capability of the ADD instruction.

3. Build and run an image that fetches its content from a remote source at build time.

4. Verify the fetched content both in the browser and from inside the running container.

🌐 ADD's "Fetch From URL" Feature ▼
This feature lets ADD pull files from a remote location — a plain zip file at a URL, or, as in this demo, a GitHub release.
ℹ️
The Setup for This Demo
A docs/ folder contains index.html, git1.html, and git2.html. That folder gets uploaded to a new public GitHub repository, and a v1.0.0 release is created from it — giving us a stable URL the Dockerfile can pull from.
📦 Prerequisite — Creating the GitHub Repository & Release ▼
ℹ️
This Step Is Optional For You
A ready-made public repository is already available for this demo's Dockerfile to reference. This section walks through how it was created, purely for extra understanding — you can skip straight to Build & Run below if you'd rather not repeat these steps yourself.
1 Create a public repository
On GitHub, create a new public repository (e.g. docker-add-fetch-url-demo), optionally initialized with a README.
2 Upload the docs/ folder
Upload the entire docs folder (containing index.html, git1.html, git2.html) and commit the change — this creates a docs/ path in the repository.
3 Create a release
Go to Releases → Create a new release. Choose (or create) the tag v1.0.0, give it a title like "Version 1.0.0", and publish it.
4 Confirm the release URL
The published release gives you a stable, versioned URL pointing at the docs folder's contents at tag v1.0.0 — that's the URL the Dockerfile references.
🧱 The Dockerfile — ADD From a GitHub Release URL ▼
Dockerfile
FROM nginx:alpine-slim

# Standard OCI labels (see Demo 4)
LABEL org.opencontainers.image.authors="Your Name"
LABEL org.opencontainers.image.title="Demo6 Dockerfile ADD Fetch From URL"
LABEL org.opencontainers.image.description="ADD instruction — fetch from a GitHub release URL"
LABEL org.opencontainers.image.version="1.0"

# ADD: fetch directly from a GitHub release URL
ADD https://github.com/<your-username>/docker-add-fetch-url-demo/releases/download/v1.0.0/docs.zip /usr/share/nginx/html/
ℹ️
What Happens at Build Time
Docker downloads whatever is at that GitHub release URL — the contents of the docs/ folder tagged v1.0.0 (index.html, git1.html, git2.html) — and places it into /usr/share/nginx/html/, exactly as if you'd run COPY on a local copy of those same files.
🔨 Build, Run & Verify ▼
1 Build the image
Run from inside the folder containing the Dockerfile:
bash
docker build -t demo6-dockerfile-add-fetch-url:v1 .
Watch the build output — you'll see Docker reach out to the GitHub URL and pull the release content down as part of the ADD step.
2 Run it
bash
docker run --name my-add-fetch-url-demo -p 8080:80 -d demo6-dockerfile-add-fetch-url:v1
3 Verify in the browser
Visit http://localhost:8080 (hard-refresh if the page looks stale) — you should see the index page with links to git1.html and git2.html, each loading correctly from a "Back to main page" link.
4 Verify from inside the container
bash
docker exec -it my-add-fetch-url-demo /bin/sh
cd /usr/share/nginx/html
ls
You should see index.html, git1.html, and git2.html — confirming the GitHub release content landed exactly where the Dockerfile said it should.
Diagram 1 — At build time, ADD reaches out over the network to grab the release content, no local copy required.
🧹 Cleanup & Summary ▼
1 Remove the container and image
bash
docker rm -f my-add-fetch-url-demo
docker rmi demo6-dockerfile-add-fetch-url:v1
✅
Summary

ADD can pull content straight from a remote URL — in this demo, a versioned GitHub release — and place it inside the image, entirely at build time.

This is exactly the kind of scenario where ADD's extra power over COPY earns its place, even though COPY remains the default choice for local files.
