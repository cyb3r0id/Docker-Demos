🏗️ The Problem With Traditional Infrastructure ▼
Definition: "Traditional infra" means deploying applications directly onto physical servers — no hypervisor, no containers. Everything is provisioned, installed, and configured by hand, server by server.
1 Provision the Hardware
Purchase physical servers and rack them in a data center.
2 Install the Operating System
Install and configure the OS on every server, one at a time.
3 Install Libraries & Dependencies
Install every library and dependency the application needs, directly on that OS.
4 Deploy Databases, App Servers & Web Servers
Manually install and configure each server role — database, application, and web — on its respective machine.
Diagram 1 — Every layer is installed and configured manually, once per physical server.
⚠️
Pain Points of Traditional Infrastructure

Time-consuming setup: install/config must be repeated on every server and every environment — dev, QA, staging, and production.

Compatibility & dependency issues: resolving library and dependency conflicts differs from server to server.

Environment inconsistency: dev, QA, staging, and prod drift apart and become hard to keep aligned.

Heavy manpower: a lot of people are needed just to manage physical servers.

Operational overhead: day-to-day hardware and OS support, patching, and scheduled downtime all take time to coordinate.

Slow onboarding: provisioning a dev environment for a new team member takes a long time.

🖥️ The Evolution to Virtual Machines ▼
ℹ️
Physical Machines Are Hard to Manage
On a physical machine you have hardware infrastructure, an operating system, then libraries/dependencies, then the application. Managing this stack across many physical machines is tough — this is where virtualization stepped in to simplify things.
Diagram 2 — A hypervisor sits on the hardware; each virtual machine gets its own OS, libraries, and app. VMs can be cloned to create more machines quickly.
🐳 Where Docker Fits In ▼
Key Idea: Docker doesn't replace physical or virtual infrastructure — it runs on top of either one. You install the Docker Engine on the operating system (physical or virtual), and the Docker Engine runs your containers.
Diagram 3 — Same Docker Engine, two possible hosts. Once it's running, containers spin up almost instantly.
✅
The Payoff — Speed & Cost
Without Docker, replicating three manually-installed applications (web, app, database) across 100 physical machines is slow and expensive, requiring heavy manpower. With Docker, you can create 100 web server containers, or 100 app server containers, in a matter of seconds — because they all come from the same underlying Docker image.
🚀 Advantages of Using Docker ▼
Flexible
Any complex application can be containerized — container tooling has advanced enough to handle almost any workload.
FLEXIBILITY
Lightweight
Containers use far fewer resources than VMs. Only the slimmed-down OS pieces the app actually needs are present — unneeded packages are excluded.
LOW OVERHEAD
Portable
Build locally, deploy to the cloud, and run anywhere. The container behaves the same regardless of the host.
RUN ANYWHERE
Loosely Coupled
You can replace or upgrade one container without affecting the others running alongside it.
INDEPENDENT
Highly Scalable
Create one container, then generate hundreds of replicas from the same Docker image with simple commands.
SCALES FAST
Highly Secure
Containers provide isolation without extra setup, and a minimal OS footprint means a smaller attack surface.
ISOLATED
ℹ️
Why Scaling Works So Well
A single Docker image can be used to create hundreds of containers — each one a replica of the same image. That's the foundation for the scalability advantage, and it's the topic of the next lecture: what is a container, and what is a Docker image?

📋 Summary & Coming Up Next ▼
✅
Summary — The Core Mental Model

Traditional infra = everything installed by hand, per physical server — slow, inconsistent, manpower-heavy.

Virtual machines = a hypervisor lets you clone full OS environments faster than buying new hardware, but each VM still carries a full OS.

Docker = the Docker Engine sits on top of either physical or virtual infrastructure, and creates lightweight, portable, scalable, secure containers from a single image — in seconds, not hours.
